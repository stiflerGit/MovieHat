# Go Testing Skill

## DO NOT write tests just to make them pass
Purpose of the tests is to identify bugs. So it's good if you write a test which is able to identify an unexpected behavior in the code you are writing the test for.
Example:
```go
func normalize(vector []float64) []float64{
	sum := 0
	for _, v := range vector {
		sum+=v
	}

	for i := range vector {
		vector[i] /= sum // sum can be 0 -> panic
	}

	return vector
}
```
Here you should write a test case for which the sum is 0

## ALWAYS Generate tests stubs with gotests
gotests is a tool to generate go test stubs. It generate tests following TDD format.
If gotests is not installed propose the user to install it running `go install github.com/cweill/gotests/gotests@latest`
Usage of gotests:
  -all    	    generate tests for all functions and methods
  -exported     generate tests for exported functions and methods. Takes precedence over -only and -all
  -only string  regexp. generate tests for functions and methods that match only. Takes precedence over -all
  -parallel     enable generating parallel subtests using the Go 1.7 feature
  -w	          write output to (test) files instead of stdout
typical usage:
gotests -all -w internal/auth -> generate a test file per each file under internal/auth package
gotests -all -w internal/auth/auth.go -> generate a test file for the file auth.go
gotests -only MyFunc -w internal/auth/auth.go -> if test file is present -> append new Test Function to the file 
                                              -> it test file not present -> create test file for the file auth.go containing only the TestFunction for MyFunc

## Remove the fields struct generated by gotests
Usually, you don't test a struct by forcing internal values. This is because you might force a state which could never happen
by using struct methods. That's why you should remove `fields` struct and use the constructor of the struct if available.
If not available and the struct is supposed to be populated manually then you can leave `fields` struct
## Generate mocks of interfaces using mockgen
Do not create structs in tests that implement an interface. It's better to use mockgen tool.
If mockgen tool is not installed propose the user to install by running `go install go.uber.org/mock/mockgen@latest` or install it as a tool with `go get -tool go.uber.org/mock/mockgen@latest`. Then you can run mockgen alternatively with `mockgen ...` or `go tool mockgen ...`.
However prefer add a go generate comment before the interface so mocks are automatically generated using `go generate` command.
Example:
```
//go:generate go tool mockgen -package mock -destination mocks/store.go . Store
type Store interface {
```
put the generated code under `mocks/` folder

mockgen has modes:
Source mode: mockgen -source=foo.go [flags]
Package mode: mockgen <import-path|.> <Iface1,Iface2> [flags]
  Example: mockgen database/sql/driver Conn,Driver

Most useful flags:
  - source: input file (enables source mode)
  - destination: output file (default stdout)
  - package: generated package name
  - imports: explicit imports (source mode)
  - mock_names: Interface=MockName overrides
  - typed: type-safe Return/Do/DoAndReturn
  - self_package: helps avoid import cycles
  - write_generate_directive: add //go:generate
  
## Add mocks to testcase struct if needed
If the test needs mocks you can add those as field of the testcase.
Example:
testcases := []struct{
  name string
  mocks func(...) ...
  args args
  want ...
  wantErr ...
}{
  ...
  mocks: func(...) ... {
    storeMock := storage.NewMock()
    ....
  }
}
## DO NOT add functions in which you make tests as fields of the testcase struct
Example:
testcases := []struct{
  name string
  act func(t *testing.T, ...)
  args args
  want ...
  wantErr ...
}{
  ...
  act: func(t *testing.T, ...)  {
    assert.Equal(...)
  }
}

## Prefer table-driven tests over many near-duplicate test functions.
Once you have stubs generated by gotests you can add test cases. 
## Use test functions instead of table-driven tests when you need to test a flow.
There are some exceptions to TDD. Take string builder as an example.
If you want to test string builder `String` function you cannot do that before calling `WriteString`.
In those cases it's better to have a single Test function. 
Example:
```go
func TestStringBuilder_String(t *testing.T) {
	b := strings.Builder{}
	b.WriteString("alpha")
	assert.Equal(t, "alpha", b.String())
	b.WriteString("beta")
	assert.Equal(t, "alphabeta", b.String())
	b.WriteString("gamma")
	assert.Equal(t,"alphabetagamma", b.String())
}
```
## Use assert and require packages from github.com/stretchr/testify for doing assertions
## Use small builders/helpers that take primitive inputs; avoid verbose setup in each case.
## Test must be understandable by just watching the test function.
If helpers are too complex it's hard to understand what the function is testing. So helpers must be self-explanatory and should not force the developer to go through the helper implementation in order to understand the test where the helper is used.
## Keep comments short: explain only non-obvious math or intent.
## Avoid banners, long doc comments, repeated loops, and custom helpers that do not remove duplication.
## Use t.Context() instead of context.Background(). t is *testing.T
